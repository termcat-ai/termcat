# 使用 Claude 命令行出现自动滚动到顶部的问题修复方案

> 关联 Issue: https://github.com/anthropics/claude-code/issues/826
> 日期: 2026-03-19

---

## 1. 问题描述

在 TermCat 的 xterm.js 终端中运行 Claude Code CLI 时，当对话内容较长（超过终端可见行数）后，终端会频繁自动滚动到顶部再跳回底部，产生严重的闪烁/频闪效果，影响使用体验。

**现象**：
- 对话历史较长时，终端 viewport 反复跳到顶部再回底部
- 流式输出时每秒闪烁多次（频闪效果）
- iTerm2 现在基本不出现滚动到顶，但会闪烁一下（立即滚回底部）

---

## 2. 根因分析

### 2.1 Claude Code 使用 Ink 渲染 TUI

Claude Code 使用 [Ink](https://github.com/vadimdemedes/ink)（React for CLI）来渲染终端 UI。Ink 的渲染逻辑在 `ink/src/ink.tsx` 中。

### 2.2 Ink 的全屏重绘策略

当输出内容高度 **>= 终端行数** 时（长对话就会触发），Ink 不再使用逐行 erase + rewrite，而是切换为 `clearTerminal` 全屏重绘：

```typescript
// ink/src/ink.tsx ~line 196
if (this.lastOutputHeight >= this.options.stdout.rows) {
    this.options.stdout.write(
        ansiEscapes.clearTerminal + this.fullStaticOutput + output
    );
}
```

`ansiEscapes.clearTerminal` 定义在 [ansi-escapes/base.js](https://github.com/sindresorhus/ansi-escapes) 中：

```javascript
// 非 Windows 平台
clearTerminal = `${eraseScreen}${ESC}3J${ESC}H`
// 展开为：
// \x1b[2J   ← ED 2: 清除整个屏幕
// \x1b[3J   ← ED 3: 清除 scrollback 缓冲区（罪魁祸首！）
// \x1b[H    ← CUP: 光标移到 (0,0) 左上角
```

关联 Ink issue: https://github.com/vadimdemedes/ink/issues/359

### 2.3 滚动到顶部的完整过程

每次 Ink 重新渲染（流式输出 token、状态栏更新、工具调用状态变化等）都会执行：

```
1. \x1b[2J        → 清除屏幕显示内容
2. \x1b[3J        → 清除 scrollback 缓冲区 ← 这会导致 viewport 位置丢失
3. \x1b[H         → 光标移到 (0,0) ← viewport 跟随光标跳到最顶部
4. 重写 fullStaticOutput + output → 内容从顶部开始逐步填充
5. 填充完毕 → viewport 回到底部
```

在流式输出时，这个过程每秒发生多次 → 用户看到频繁的 **scroll-to-top 闪烁**。

## 3. 各终端表现差异

| 终端 | 表现 | 原因 |
|------|------|------|
| **xterm.js (TermCat)** | 严重闪烁，频繁 scroll-to-top | 忠实处理所有转义序列，`\x1b[3J` 清除 scrollback，`\x1b[H` 光标归零时 viewport 跟随 |
| **VSCode/Cursor 内置终端** | 严重闪烁，甚至卡死 | 同样基于 xterm.js，且还有额外的 UI 层开销 |
| **iTerm2** | 基本不闪，偶尔轻微闪烁 | 拦截 `\x1b[3J`（弹出 "A control sequence attempted to clear scrollback history" 提示），viewport 不跟随光标跳转 |
| **Ghostty** | 不闪 | GPU 渲染 + vsync，快速的清屏-重写合并为一帧，用户看不到中间状态 |
| **Warp** | 不闪 | 自研渲染引擎，不是传统终端模拟器，不受这些转义序列影响 |
| **tmux** | copy-mode 下不闪 | 提供虚拟终端层，copy-mode 下不响应底层清屏操作 |
| **macOS Terminal** | 轻微闪烁 | Core Text 渲染器处理 `\x1b[3J` 比较保守 |
| **GNOME Terminal** | 闪烁 | 忠实处理转义序列 |
| **Windows Terminal** | 独立窗口时较好 | GPU 加速渲染 |

### 核心区别

**不闪的终端**有以下一种或多种特性：
1. **拦截 `\x1b[3J`**：不执行清除 scrollback 的操作（iTerm2）
2. **帧合并渲染**：GPU 渲染将多次快速写入合并为一帧显示（Ghostty、Warp）
3. **viewport 独立于光标**：光标移到 (0,0) 时 viewport 不跟随（iTerm2）

**闪烁的终端**（包括 xterm.js）：
- 忠实处理每条转义序列
- viewport 跟随光标位置
- 每次 write 都立即反映到渲染

## 4. 修复方案

### 4.1 方案一：过滤 `\x1b[3J`（推荐，最小改动）

在 `XTermTerminal.tsx` 的 `backend.onData` 回调中，过滤掉 `\x1b[3J`（ED 3 - Erase Scrollback）：

```typescript
// 在 filterFontChangingSequences 函数中或 onData 回调中添加：

// 过滤 Ink clearTerminal 中的 \x1b[3J（清除 scrollback 缓冲区）
// 这是 Claude Code CLI 使用 Ink 渲染时导致 scroll-to-top 闪烁的罪魁祸首
// iTerm2 也是通过拦截这个序列来避免闪烁的
filteredData = filteredData.replace(/\x1b\[3J/g, '');
```

**效果**：
- `\x1b[2J`（清屏）+ `\x1b[H`（光标归零）仍然生效，Ink 的重绘逻辑正常工作
- scrollback 历史不会被销毁，viewport 不会跳到顶部
- 与 iTerm2 的处理方式一致

**风险**：低。`\x1b[3J` 在正常 SSH 终端使用中极少出现（只有 `clear` 命令和 Ink 的 clearTerminal 会用到），过滤后不影响常规终端操作。

### 4.2 方案二：viewport 位置保持（进阶，更彻底）

如果方案一后仍有轻微闪烁（因为 `\x1b[2J\x1b[H` 仍会让屏幕清空再重写），可以进一步：

```typescript
// 在 terminal.write() 前后保存/恢复 viewport 位置
const viewport = (terminal as any)._core?.viewport;
const scrollTop = viewport?.scrollArea?.scrollTop;

terminal.write(batch);

// 如果检测到全屏重绘模式，恢复 viewport 位置
if (scrollTop !== undefined && batch.includes('\x1b[2J')) {
    requestAnimationFrame(() => {
        if (viewport?.scrollArea) {
            viewport.scrollArea.scrollTop = scrollTop;
        }
    });
}
```

### 4.3 方案三：检测 Ink 重绘模式，合并为单帧

检测 `\x1b[2J.*\x1b[H` 模式，将整个重绘序列缓冲起来，在下一个 `requestAnimationFrame` 中一次性写入，避免用户看到中间状态。

```typescript
// 伪代码
if (data.includes('\x1b[2J') && data.includes('\x1b[H')) {
    // 这是 Ink 的全屏重绘，缓冲到下一帧一次性写入
    inkRewriteBuffer += data;
    if (inkRewriteRaf === null) {
        inkRewriteRaf = requestAnimationFrame(() => {
            terminal.write(inkRewriteBuffer);
            inkRewriteBuffer = '';
            inkRewriteRaf = null;
        });
    }
} else {
    terminal.write(data);
}
```

> 注意：当前 TermCat 已有 rAF 批量写入机制（P6），方案三的逻辑可以与现有机制合并。

---

## 5. 建议实施顺序

1. **先实施方案一**（过滤 `\x1b[3J`），一行代码修复，效果立竿见影
2. 测试观察是否还有残余闪烁
3. 如仍有问题，再叠加方案二或方案三
