# 跳板机（Jump Host）技术方案

## 1. 概述

termcat 支持通过跳板机（堡垒机）连接目标主机。用户在主机配置中选择"跳板机"连接类型，填写目标主机地址，即可通过跳板机中转访问内网服务器。

### 数据流

```
termcat (本地) ──SSH──▶ 跳板机 ──tunnel/ssh──▶ 目标主机
```

### 涉及文件

| 文件 | 职责 |
|---|---|
| `src/main/ssh-service.ts` | 主进程 SSH 连接核心，跳板机三级 fallback 逻辑 |
| `src/main/ssh-config-parser.ts` | 解析 `~/.ssh/config`，按主机匹配应用配置 |
| `src/services/sshService.ts` | 渲染进程 SSH 服务，组装跳板机配置发送到主进程 |
| `src/services/api.ts` | API 服务，传递 `connection_type` / `target_host` |
| `termcat_server/internal/models/host.go` | Host 模型，`ConnectionType` + `TargetHost` 字段 |
| `termcat_server/internal/api/host.go` | Host API，CRUD 支持跳板机字段 |

## 2. 全栈数据流

### 2.1 数据库模型（Go）

```go
// internal/models/host.go
type Host struct {
    // ...
    ConnectionType string `gorm:"size:20;default:direct" json:"connection_type"` // direct, jump
    TargetHost     string `gorm:"size:255" json:"target_host"`                   // 跳板机模式下的目标主机地址
}
```

GORM AutoMigrate 会自动添加新列，无需手动迁移。

### 2.2 API 层（Go）

`CreateHostRequest` 和 `UpdateHostRequest` 均包含 `ConnectionType` 和 `TargetHost` 字段。API handler 在创建/更新时读取并存储，在查询时返回。

### 2.3 前端 API 服务

```typescript
// src/services/api.ts
// getHosts() 响应映射
connectionType: host.connection_type,
targetHost: host.target_host,

// createHost() / updateHost() 请求体
connection_type: host.connectionType,
target_host: host.targetHost,
```

### 2.4 前端 SSH 服务

```typescript
// src/services/sshService.ts
const sshConfig = {
  host: host.connectionType === 'jump' ? host.targetHost : host.hostname,
  port: host.connectionType === 'jump' ? 22 : (Number(host.port) || 22),
  username: host.username,
};

if (host.connectionType === 'jump' && host.targetHost) {
  sshConfig.jumpHost = {
    host: host.hostname,       // 跳板机地址
    port: Number(host.port) || 22,
    username: host.username,   // 与目标共享凭据
  };
}

// 认证凭据同时赋给 jumpHost
if (sshConfig.jumpHost) {
  if (sshConfig.password) sshConfig.jumpHost.password = sshConfig.password;
  if (sshConfig.privateKey) sshConfig.jumpHost.privateKey = sshConfig.privateKey;
}
```

## 3. 主进程连接逻辑（三级 Fallback）

`connectViaJumpHost()` 方法实现了三级递进式 fallback 策略，以兼容不同类型的跳板机/堡垒机：

```
forwardOut (direct-tcpip)
    │
    ├─ 成功 → SSH 握手 → 成功 → 完成 ✓
    │                   └─ 失败 ─┐
    └─ 失败 ────────────────────┘
                                 ▼
                        exec proxy (nc/ncat/socat)
                            │
                            ├─ exec 通道失败 ─────┐
                            ├─ 成功 → SSH 握手 ─┐  │
                            │    └─ 成功 → 完成 ✓│  │
                            │    └─ 失败 ───────┘  │
                            └──────────────────────┘
                                                    ▼
                                          shell passthrough
                                          (ssh target_host)
                                                → 完成 ✓
```

### 3.1 Fallback 1: forwardOut（direct-tcpip 通道）

标准 SSH 端口转发，等价于 OpenSSH 的 `-L` 参数。

```typescript
jumpClient.forwardOut('127.0.0.1', 0, config.host, config.port, (err, stream) => {
  if (!err) {
    connectTargetViaStream(stream, {
      onFail: (targetErr) => tryExecProxy(targetErr.message),
    });
  } else {
    tryExecProxy(err.message);
  }
});
```

**原理**：请求跳板机的 SSH 服务端打开到目标主机的 TCP 连接，返回一个双向 stream。termcat 在此 stream 上建立第二层 SSH 连接（targetClient）。

**失败场景**：企业堡垒机通常配置 `AllowTcpForwarding no`，此时 forwardOut 会返回 `Channel open failure: open failed` 错误。

**特殊情况**：部分堡垒机接受 forwardOut 请求（不报错），但实际不转发数据——stream 立即关闭，导致 `Connection lost before handshake`。此时通过 `onFail` 回调触发下一级 fallback。

### 3.2 Fallback 2: exec proxy（nc/ncat/socat TCP 隧道）

通过 SSH exec 通道在跳板机上执行网络工具建立 TCP 隧道。

```typescript
const proxyCmd = [
  `if command -v nc >/dev/null 2>&1; then exec nc ${host} ${port};`,
  `elif command -v ncat >/dev/null 2>&1; then exec ncat ${host} ${port};`,
  `elif command -v socat >/dev/null 2>&1; then exec socat - TCP:${host}:${port};`,
  `else echo "NO_PROXY_TOOL" >&2; exit 1;`,
  `fi`,
].join(' ');
```

**原理**：在跳板机上运行 nc 等工具，将 exec stream 的 stdin/stdout 桥接到目标主机的 TCP 连接。`exec` 前缀使 nc 替换 shell 进程，消除额外开销。termcat 在此 stream 上建立 targetClient SSH 连接。

**诊断信息**：监听 exec stream 的 stderr 和 exit 事件，在 SSH 握手失败时附带 proxy 的错误输出，帮助定位问题（如 `NO_PROXY_TOOL`、nc 连接拒绝等）。

**失败场景**：堡垒机限制 exec 通道、无 nc/ncat/socat 工具、或目标主机从跳板机不可达。失败后触发下一级 fallback。

### 3.3 Fallback 3: Shell Passthrough（最终方案）

在跳板机的 shell 中直接执行 `ssh` 命令连接目标主机，与用户在 Mac 终端手动操作完全等价。

```typescript
const tryShellPassthrough = (prevErrors: string) => {
  // 将 jumpClient 存为 connection 的 client（不创建 targetClient）
  this.connections.set(connectionId, {
    id: connectionId,
    client: jumpClient,
    connected: true,
    banner: jumpBanner,
    shellPassthroughCmd: `ssh -tt -o StrictHostKeyChecking=no ${config.host}\n`,
    // ...
  });
  resolve(connectionId);
};
```

**createShell 注入**：

```typescript
// createShell() 中，shell 创建后检查 passthrough 标记
if (!isExtraShell && connection.shellPassthroughCmd) {
  setTimeout(() => {
    stream.write(connection.shellPassthroughCmd);
  }, 500);  // 等待跳板机 prompt 就绪
}
```

**原理**：
1. 将 jumpClient 直接作为连接的 SSH client 存储
2. `createShell()` 在跳板机上打开 shell（而非目标主机）
3. shell 就绪后自动注入 `ssh -tt -o StrictHostKeyChecking=no target_host`
4. 用户看到的终端输出即为目标主机的 shell

**特殊处理**：
- 跳过 MOTD 获取（堡垒机可能不支持 exec 通道，MOTD 在 shell 输出中自然显示）
- 使用跳板机的 SSH Banner（ZEGO 图案等）
- jumpClient close 时不重复调用 `client.end()`（passthrough 模式下 connection.client === jumpClient）

**用户体验**：终端依次显示跳板机 banner → 跳板机 prompt → ssh 命令 → 目标主机 banner/MOTD → 目标主机 shell，与手动在 Mac 终端中操作的体验一致。

**限制**：passthrough 模式下没有到目标主机的独立 SSH session，因此 SFTP、exec 等基于 SSH 协议的高级功能不可用。对于纯终端交互场景完全满足。

## 4. 辅助机制

### 4.1 ~/.ssh/config 解析

`ssh-config-parser.ts` 解析本地 SSH 配置文件，为跳板机和目标主机分别 resolve 配置：

```typescript
const resolvedTarget = sshConfigParser.resolve(config.host);
const resolvedJump = config.jumpHost ? sshConfigParser.resolve(config.jumpHost.host) : undefined;
```

支持的指令映射：

| SSH Config 指令 | ssh2 选项 | 说明 |
|---|---|---|
| `ForwardAgent yes` | `agentForward: true` + `agent` | Agent 转发（跳板机场景重要） |
| `IdentityFile` | `privateKey`（读取文件内容） | 仅 UI 未配密钥时使用 |
| `ServerAliveInterval` | `keepaliveInterval`（×1000 转 ms） | 保活间隔 |

优先级：UI 配置 > SSH config 文件。

### 4.2 SSH Banner 捕获

跳板机和目标主机均监听 `client.on('banner')` 事件，在 SSH 认证阶段捕获 banner 内容。`createShell()` 时将 banner 发送到终端显示。

```typescript
jumpClient.on('banner', (message: string) => { jumpBanner = message; });
targetClient.on('banner', (message: string) => { targetBanner = message; });
```

### 4.3 连接生命周期

```
jumpClient.connect()
    │
    ▼
jumpClient 'ready'
    │
    ├─ [tunnel mode] forwardOut/exec → targetClient.connect()
    │     targetClient 'ready' → connection.client = targetClient
    │     jumpClient 'close' → targetClient.end() → connection close
    │
    └─ [passthrough mode] connection.client = jumpClient
          jumpClient 'close' → connection close（不重复 end）
```

## 5. 遇到的问题与解决方案

### 5.1 Channel open failure: open failed

**现象**：`forwardOut` 返回错误 `Channel open failure: open failed`。

**原因**：企业堡垒机配置 `AllowTcpForwarding no`，禁用了 direct-tcpip 通道。

**解决**：添加 exec proxy fallback（nc/ncat/socat），在跳板机上通过 exec 通道运行网络工具建立 TCP 隧道。

### 5.2 Connection lost before handshake（exec proxy 路径）

**现象**：exec proxy 启动后，targetClient SSH 握手立即失败。

**原因**：
1. 最初实现中有 1.5 秒延迟用于探测 nc 是否可用——nc 连接目标后，SSH 服务端立即发送版本字符串，但 1.5 秒内无人读取，服务端超时断开。
2. 部分堡垒机虽然 exec 通道可打开，但实际不执行命令或限制网络工具使用。

**解决**：
1. 移除延迟，exec stream 创建后立即调用 `connectTargetViaStream()`。
2. 将 exec proxy 的失败也接入 fallback 链，自动降级到 shell passthrough。

### 5.3 Connection lost before handshake（forwardOut 路径）

**现象**：`forwardOut` 回调无 error（表面成功），但后续 SSH 握手立即失败。

**原因**：堡垒机接受了 direct-tcpip 请求但实际不转发数据，stream 被立刻关闭。

**解决**：`connectTargetViaStream` 增加 `onFail` 回调机制——forwardOut 路径传入回调函数，SSH 握手失败时不直接 reject，而是触发 exec proxy fallback。三级 fallback 链自此形成：

```
forwardOut 成功但握手失败 → exec proxy → exec 也失败 → shell passthrough
```

### 5.4 跳板机配置不持久化

**现象**：保存为跳板机类型，重新打开后显示为直连。

**原因**：全栈缺失——Go 模型无 `ConnectionType`/`TargetHost` 字段，API 不传递，前端不映射。

**解决**：四层同步添加：
1. Go Host 模型添加 `ConnectionType` + `TargetHost` 字段
2. API CreateHostRequest / UpdateHostRequest 添加字段
3. API handler 读写新字段
4. 前端 `api.ts` 请求/响应映射新字段

### 5.5 MOTD/Banner 不显示

**现象**：Mac 终端显示 ZEGO ASCII Art 和欢迎信息，termcat 不显示。

**原因**：SSH Banner 在认证阶段通过 SSH 协议发送（`banner` 事件），而非 shell 输出。ssh2 的 `Client` 需要显式监听此事件。

**解决**：在 `Client` 上添加 `banner` 事件监听，缓存 banner 内容，`createShell()` 时发送到终端。

### 5.6 登录后出现重复 Prompt

**现象**：`[dum@zego ~]$ [dum@zego ~]$` 登录约 1 秒后出现双重提示符。

**原因**：`XTermTerminal.tsx` 中有 `setTimeout` 在 2 秒后注入 `export TERM=xterm-256color\n`，产生额外输出和 prompt。

**解决**：删除该 setTimeout。PTY 请求（`client.shell({ term: 'xterm-256color' })`）已通过 SSH 协议设置 TERM 环境变量，无需额外 export。

### 5.7 远程脚本中文乱码（UnicodeEncodeError）

**现象**：跳板机上运行含中文输出的 Python 脚本报 `UnicodeEncodeError: 'ascii' codec can't encode characters`。

**原因**：ssh2 的 `shell()` 默认不设置 LANG 环境变量，导致远程 shell 的 locale 为 C/POSIX（ASCII only）。Mac 终端的 SSH 自动发送 `LANG`。

**解决**：`createShell()` 时在 `shell()` 的第二个参数（ShellOptions）中传递 `{ env: { LANG: process.env.LANG || 'en_US.UTF-8' } }`。

注意 ssh2 的 `shell()` 签名为 `shell(pseudoTtyOptions, shellOptions, callback)`，`env` 属于第二个参数 `ShellOptions`，不是第一个参数 `PseudoTtyOptions`。
