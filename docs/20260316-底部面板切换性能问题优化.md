# 底部面板显示时切换终端 Tab 卡顿问题

**日期**: 2026-03-16
**状态**: 已修复

优化内容

层1: CSS content-visibility: hidden (对 tab 切换影响最大)

原理: 非活跃终端 tab 的面板（底部面板、左/右侧栏）由浏览器跳过布局和绘制，显著减少合成器在切换 z-index 时的工作量。

- App.tsx: 非活跃 tab 的 wrapper div 添加 data-tab-inactive 属性
- TerminalView.tsx: 底部面板添加 tv-bottom-panel 类名，左/右侧栏添加 tv-side-panel 类名
- index.css: 新增 CSS 规则，对 [data-tab-inactive] 下的面板容器应用 content-visibility: hidden

效果：3 个 tab 打开时，非活跃 tab 的底部面板和侧栏内容不参与浏览器绘制，等于只渲染 1 个 tab 的面板。

层2: React.memo + useCallback (对 tab 内部交互性能)

- TreeNode: 添加 React.memo，防止递归树的全量重渲染
- FileTreePanel: 添加 React.memo，文件列表变化时目录树不重渲染
- FileListPanel: 添加 React.memo，目录树变化时文件列表不重渲染
- FileBrowserPanel: 13 个 handler 函数用 useCallback 包裹，配合 currentPathRef / filesRef / selectedFilesRef 避免依赖频繁变化的值

进一步优化建议（如果还卡）

- 文件列表虚拟化: 用 react-virtuoso（项目已有该依赖）替换 <table> 的全量渲染，100 行文件只渲染可见的 ~15 行
- 目录树深度限制: getDirectoryTree('/', 3) 在文件系统庞大时可能返回数百个节点，可考虑按需展开


## 问题描述

打开 3 个终端 Tab，显示底部面板（文件浏览器）后，切换终端 Tab 时明显卡顿。隐藏底部面板后切换流畅。

## 根因分析

### 架构背景

TerminalView 采用 **z-index 层叠** 方式切换 Tab——所有终端始终挂载在 DOM 中，仅通过 z-index 控制可见性，避免 xterm canvas 重绘闪烁。`TerminalView` 外层有 `React.memo`，故意不比较 `isActive`，确保切换时不触发 React 重渲染。

### 瓶颈定位

问题不在 React 重渲染（memo 正确阻止了），而在 **浏览器合成器（Compositor）** 层面：

```
无底部面板时：
  3 个 tab × ~100-200 DOM 节点（xterm canvas + 工具栏）= ~600 节点
  → 合成器切换 z-index 很快

有底部面板时：
  3 个 tab × ~500-1000 DOM 节点（xterm + 文件树 + 文件列表表格 + lucide 图标）= ~3000 节点
  → 合成器处理大型图层，z-index 切换变慢
```

具体 DOM 来源（每个 Tab）：
- **文件目录树**：`find / -maxdepth 3 -type d` 可返回 50-200 个节点，每个节点递归渲染 `TreeNode`
- **文件列表表格**：每个目录 50-100+ 文件 × 8 列 × lucide SVG 图标
- **其他底部面板**：命令库、传输管理器同时挂载（通过 `display: none` 隐藏）

3 个 Tab 全部同时挂载 = **非活跃 Tab 的底部面板白白消耗浏览器资源**。

### 附带问题：组件内部无优化

`FileBrowserPanel` 的子组件（`TreeNode`、`FileTreePanel`、`FileListPanel`）均无 `React.memo`，handler 函数也未 `useCallback`。虽然 TerminalView 的 memo 在 Tab 切换时阻止了重渲染，但 Tab 内部的任何状态变化（如点击目录树节点）都会导致所有子组件全量重渲染。

## 修复方案

### 层 1：CSS `content-visibility: hidden`（解决 Tab 切换卡顿）

**核心思路**：让浏览器跳过非活跃 Tab 中面板的布局和绘制。

#### 改动

| 文件 | 改动 |
|------|------|
| `src/renderer/App.tsx` | 非活跃 Tab 的 wrapper div 添加 `data-tab-inactive` 属性 |
| `src/components/TerminalView.tsx` | 底部面板容器加 `tv-bottom-panel` 类名，左/右侧栏加 `tv-side-panel` 类名 |
| `src/renderer/styles/index.css` | 新增 CSS 规则 |

#### CSS 规则

```css
[data-tab-inactive] :is(.tv-bottom-panel, .tv-side-panel) {
  content-visibility: hidden;
}
```

#### 原理

- `content-visibility: hidden` 告诉浏览器跳过该子树的布局和绘制
- 面板容器因有显式尺寸（`width` / `height` style）仍保留占位，不影响整体布局
- Electron 28 使用 Chrome ~120，完全支持 `content-visibility`
- **无 React 侵入**：不改变组件逻辑，不破坏 TerminalView 的 memo 契约
- 父级 div 在 App.tsx 的 `map()` 中，不受 memo 影响，`data-tab-inactive` 随 `isActive` 正确更新

#### 效果

3 个 Tab 打开时，浏览器只绘制 **1 个 Tab** 的底部面板和侧栏，其余 2 个完全跳过。等效 DOM 工作量从 ~3000 降至 ~1000。

### 层 2：React.memo + useCallback（解决 Tab 内部交互性能）

#### 组件 memo 化

| 组件 | 文件 | 作用 |
|------|------|------|
| `TreeNode` | `FileTreePanel.tsx` | 递归组件 memo 化，展开/折叠一个节点时其他节点不重渲染 |
| `FileTreePanel` | `FileTreePanel.tsx` | 文件列表变化时目录树不重渲染 |
| `FileListPanel` | `FileListPanel.tsx` | 目录树变化时文件列表不重渲染 |

#### Handler 函数 useCallback 化

`FileBrowserPanel.tsx` 中 13 个传给子组件的 handler 全部 `useCallback` 包裹：

| Handler | 技巧 |
|---------|------|
| `handleTreeNodeClick` | `useCallback([loadFiles])` |
| `toggleTreeNode` | `useCallback([])` — 使用 `setDirectoryTree(prev => ...)` 函数式更新 |
| `handleRefreshFiles` | 通过 `currentPathRef` 读取最新路径，避免 `currentPath` 作为依赖 |
| `handleGoToParent` | 同上 |
| `handleFileDoubleClick` | 同上 |
| `handleFileDragStart` | `useCallback([connectionId])` |
| `handleDownload` | `useCallback([connectionId, t])` |
| `handleUploadClick` | `useCallback([connectionId, loadFiles, t])` |
| `handleDownloadClick` | 通过 `selectedFilesRef` + `filesRef` + `currentPathRef` 读取最新状态 |
| `toggleFileSelection` | `useCallback([])` — 函数式更新 |
| `toggleSelectAll` | 通过 `filesRef` 读取最新文件列表 |
| `handleDragOver` / `handleDragLeave` | `useCallback([])` |
| `handleTreeContextMenu` / `handleListContextMenu` / `handleListBlankContextMenu` | `useCallback([])` — 通过 ref 读取 currentPath |
| `handleListDrop` | `useCallback([connectionId])` — 通过 ref 读取 currentPath |

**Ref 模式说明**：`currentPath`、`files`、`selectedFiles` 等频繁变化的状态通过 `useRef` + 同步赋值保持最新值，使 `useCallback` 不依赖这些值，保证回调引用稳定。

```typescript
const currentPathRef = useRef(currentPath);
currentPathRef.current = currentPath;

const handleGoToParent = useCallback(() => {
  const cp = currentPathRef.current;
  if (cp === '/') return;
  const parentPath = cp.substring(0, cp.lastIndexOf('/')) || '/';
  loadFiles(parentPath);
}, [loadFiles]); // 不依赖 currentPath，引用稳定
```

## 进一步优化建议

如果优化后仍有卡顿，可考虑：

1. **文件列表虚拟化**：用 `react-virtuoso`（项目已有该依赖）的 `TableVirtuoso` 替换当前 `<table>` 全量渲染。100 行文件只渲染可见的 ~15 行，单个 Tab 减少 ~500 DOM 节点。

2. **目录树按需加载**：当前 `getDirectoryTree('/', 3)` 在文件系统庞大时可能返回数百个节点。改为初始只加载第一层，展开节点时再异步加载子目录。

3. **lucide 图标缓存**：文件列表中每行的 `Folder` / `FileText` / `Download` 图标是独立 SVG 渲染。可考虑将高频图标改为 CSS sprite 或内联 SVG 常量，减少组件实例数。
