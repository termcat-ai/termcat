# 桌面本地终端 AI 等能力扩展实现方案（第二期）

> 日期：2026-03-19
> 前置文档：[第一期方案](./20260319-桌面本地终端方案.md)
> 状态：设计中

---

## 1. 背景

第一期已实现基于 `ITerminalBackend` 抽象层的本地终端核心功能（node-pty + Shell 检测 + Tab 混合显示）。但本地终端目前仅是"纯终端"，AI Agent、文件浏览、系统监控等周边能力均无法使用。

本期目标：通过**面向对象抽象**和**策略模式**，在不改动已有 SSH 逻辑的前提下，让 AI Agent、文件浏览器、系统监控三个子系统统一支持本地终端。

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| **接口隔离** | 每个子系统定义独立抽象接口，SSH 与本地各自实现，互不耦合 |
| **策略模式** | 上层通过接口调用，由工厂类根据 connectionType 注入具体实现 |
| **零改动现有 SSH** | 所有 SSH 实现文件不修改，新增本地实现文件与之平行 |
| **插件感知** | 插件通过 `ConnectionInfo` 感知连接类型，自动选择正确的服务实现 |
| **渐进式** | 四个子系统可独立开发、独立上线，无强依赖 |

---

## 3. 总体架构

### 3.1 三层分层模型

系统按职责分为三层，每层通过抽象接口解耦：

```
┌─────────────────────────────────────────────────────────┐
│                    应用层（业务功能）                       │
│                                                         │
│   FileBrowserService      MonitorService                │
│   （文件浏览）              （系统监控）                    │
│                                                         │
│   依赖能力层接口，不感知底层是 SSH 还是 Local              │
└──────────────┬─────────────────────┬────────────────────┘
               │ 依赖                │ 依赖
┌──────────────▼─────────────────────▼────────────────────┐
│                    能力层（基础能力）                       │
│                                                         │
│   IHostConnection (host 连接总入口)                       │
│   ├── type: 'ssh' | 'local'                             │
│   ├── connect() / disconnect()     ← 连接生命周期        │
│   ├── terminal: ITerminalBackend   ← 终端 I/O           │
│   ├── executor: ICommandExecutor   ← 命令执行            │
│   └── transfer?: IFileTransfer     ← 文件传输(SSH 有)    │
│                                                         │
│   SSHHostConnection    LocalHostConnection               │
│   (组合 SSH 实现)       (组合 Local 实现)                  │
└──────────────┬─────────────────────┬────────────────────┘
               │                     │
┌──────────────▼─────────────────────▼────────────────────┐
│                    传输层（底层通信）                       │
│                                                         │
│   ssh-service           local-pty-service                │
│   file-transfer-service  local-fs-service                │
│   tunnel-service         local-monitor-service           │
│                                                         │
│   （未来迁移至 src/core/）                                │
└─────────────────────────────────────────────────────────┘
```

### 3.2 IHostConnection — host 连接统一入口

上层（TerminalView、插件、Hook）仅持有 `IHostConnection` 引用，通过它获取所有能力，不感知底层是 SSH 还是 Local：

```
            IHostConnection (新增接口)
            ┌──────────────────────────────┐
            │ type: 'ssh' | 'local'        │
            │ connect(): Promise<void>     │
            │ disconnect(): Promise<void>  │
            │ terminal: ITerminalBackend   │
            │ executor: ICommandExecutor   │
            │ transfer?: IFileTransfer     │
            └──────────────┬───────────────┘
                           │
              ┌────────────┼────────────┐
              ▼                         ▼
    SSHHostConnection          LocalHostConnection
    ┌──────────────────┐    ┌──────────────────┐
    │ terminal:        │    │ terminal:        │
    │  SSHTerminal     │    │  LocalTerminal   │
    │  Backend         │    │  Backend         │
    │ executor:        │    │ executor:        │
    │  ElectronShell   │    │  LocalShell      │
    │  Executor        │    │  Executor        │
    │ transfer:        │    │ transfer: null   │
    │  SFTPTransfer    │    │  (本地无需)       │
    └──────────────────┘    └──────────────────┘
```

**关键设计**：
- `IHostConnection` 是**组合模式**，聚合同层的多个能力接口
- 应用层（FileBrowser、Monitor）从 `IHostConnection` 取 `executor` 执行命令，或本地模式下直接用 `fs` API
- `IHostConnection` 不包含 FileBrowser / Monitor — 它们是应用层，层级更高

### 3.3 连接类型路由

```
TerminalView 创建连接
       │
       ▼
HostConnectionFactory.create(host)
       │
       ├── connectionType === 'local'
       │     → new LocalHostConnection()
       │
       └── connectionType === 'ssh'
             → new SSHHostConnection(sshConfig)
       │
       ▼
IHostConnection 实例
       │
       ▼
TerminalView 持有 connection 引用
       │
       ├── connection.terminal → 传给 XTermTerminal
       ├── connection.executor → 传给 useAIAgent
       └── connection → 传给 ConnectionInfo → 插件路由
```

### 3.4 ConnectionInfo 扩展

`ConnectionInfo` 新增 `connectionType` 字段，插件通过类型字段路由，不再依赖 connectionId 字符串前缀：

```
ConnectionInfo {
  connectionId: string
  connectionType: 'ssh' | 'local'    // 新增
  hostname: string
  isVisible: boolean
  isActive: boolean
  language: string
}
```

## 4. 子系统一：AI Agent 本地执行器

### 4.1 问题分析

当前 AI Agent 通过 `ICommandExecutor` 执行命令，已有实现 `ElectronShellExecutor` 依赖 SSH IPC。本地终端需要一个新的执行器，通过 Local PTY IPC 通道执行命令。

### 4.2 设计思路

```
                    ICommandExecutor (已有接口)
                    ┌────────────────────────┐
                    │ initialize()           │
                    │ execute(cmd) → Result  │
                    │ cleanup()              │
                    │ isReady()              │
                    └──────────┬─────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
    BaseShellExecutor    (其他 Executor)
    ┌─────────────────┐
    │ 标记注入         │
    │ Heredoc 修复     │
    │ 交互检测         │
    │ 超时管理         │
    │ sudo 密码处理    │
    └──────┬──────────┘
           │
    ┌──────┼──────────────┐
    ▼                     ▼
ElectronShell         LocalShell
Executor (已有)       Executor (新增)
┌──────────────┐   ┌──────────────┐
│ setupShell() │   │ setupShell() │
│  → SSH IPC   │   │  → PTY IPC  │
│ writeRaw()   │   │ writeRaw()  │
│  → sshWrite  │   │  → ptyWrite │
│ onShellData()│   │ onShellData()│
│  → onShellData│  │  → onPtyData│
└──────────────┘   └──────────────┘
```

**核心点**：`LocalShellExecutor` 继承 `BaseShellExecutor`，只需实现 3 个抽象方法：

| 方法 | SSH 实现 | 本地实现 |
|------|---------|---------|
| `setupShell()` | `sshCreateShell()` 创建 SSH shell | `localTerminal.create()` 创建 PTY |
| `writeRaw(data)` | `sshShellWrite(connId, data)` | `localTerminal.write(ptyId, data)` |
| `onShellDataSetup(cb)` | `onShellData()` 过滤 connId | `localTerminal.onData()` 过滤 ptyId |

`BaseShellExecutor` 中的所有通用逻辑（命令标记注入、输出解析、heredoc 修复、超时管理、交互式提示检测、sudo 密码处理）**完全复用**，零重复代码。

### 4.3 流程

```
用户在本地终端的 AI 面板输入 prompt
  │
  ▼
useAIAgent 检测 connectionType === 'local'
  │
  ▼
创建 LocalShellExecutor（传入当前 PTY 的 ptyId）
  │
  ▼
AIAgent.setExecutor(localShellExecutor)
  │
  ▼
AIAgent.ask(prompt, mode='agent')
  │
  ▼
Agent Server 返回 EXECUTE_REQUEST
  │
  ▼
AIAgent 调用 localShellExecutor.execute(command)
  │
  ▼
LocalShellExecutor:
  1. BaseShellExecutor.execute() 注入标记
  2. writeRaw() → IPC: localTerminal.write(ptyId, wrappedCmd)
  3. onShellData 回调收集输出
  4. 检测标记 → 提取 exitCode + output
  │
  ▼
返回 CommandResult { success, output, exitCode }
  │
  ▼
AIAgent 发送 CONFIRM_EXECUTE 给 Server
  │
  ▼
继续下一步 / 完成
```

### 4.4 重点问题

| 问题 | 解决方案 |
|------|---------|
| AI 执行器与用户终端共用 PTY？ | **独立模式**：创建第二个 PTY 专用于 AI（类似 SSH 的 `__ai_shell` 模式），避免与用户输入冲突 |
| 本地终端无需 sudo 密码？ | macOS/Linux 的 sudo 仍需密码，`BaseShellExecutor` 已有密码注入逻辑，直接复用 |
| Code 模式（Claude SDK）的工具执行？ | Code 模式的 `remote_terminal_proxy` 走 EXECUTE_REQUEST → 同样路由到 `LocalShellExecutor` |
| Shell 环境差异？ | 本地 Shell 环境变量与 SSH 不同，但 `BaseShellExecutor` 的标记机制与环境无关 |

## 5. 子系统二：本地文件浏览器

### 5.1 问题分析

当前 `FileBrowserService` 所有操作通过 `window.electron.sshExecute()` 执行远程 Shell 命令实现。本地终端需要一个使用 Node.js `fs` API 的本地实现。

### 5.2 设计思路

引入 `IFileBrowserService` 抽象接口，SSH 和本地各自实现：

```
            IFileBrowserService (新增接口)
            ┌──────────────────────────┐
            │ listFiles(path)          │
            │ getFileContent(path)     │
            │ writeFileContent(p, c)   │
            │ getDirectoryTree(p, d)   │
            │ rename(dir, old, new)    │
            │ deleteFile(dir, name)    │
            │ mkdir(dir, name)         │
            │ createFile(dir, name)    │
            │ chmod(dir, name, octal)  │
            │ packFiles(dir, names)    │
            └────────────┬─────────────┘
                         │
            ┌────────────┼────────────┐
            ▼                         ▼
  SSHFileBrowser              LocalFileBrowser
  Service (现有重命名)         Service (新增)
  ┌────────────────┐       ┌────────────────┐
  │ sshExecute()   │       │ fs.readdir()   │
  │ 解析 ls 输出    │       │ fs.stat()      │
  │ base64 编码     │       │ fs.readFile()  │
  └────────────────┘       │ fs.writeFile() │
                           │ fs.rename()    │
                           │ fs.rm()        │
                           └────────────────┘
```

### 5.3 流程

```
FileBrowserPanel 收到 ConnectionInfo
  │
  ▼
检查 connectionType
  │
  ├── 'ssh' → new SSHFileBrowserService(connectionId)
  └── 'local' → new LocalFileBrowserService()
  │
  ▼
FileBrowserPanel 调用 service.listFiles(path)
  │
  ├── SSH: sshExecute('ls -alh ...') → 解析输出
  └── Local: fs.readdir() + fs.stat() → 构造 FileItem[]
  │
  ▼
UI 渲染文件列表（两种实现返回相同的 FileItem[] 结构）
```

### 5.4 本地实现要点

| 操作 | SSH 实现 | 本地实现 |
|------|---------|---------|
| 列出文件 | `ls -alh` + 解析 | `fs.readdir()` + `fs.stat()` |
| 读取文件 | `cat \| base64` | `fs.readFile()` |
| 写入文件 | `echo \| base64 -d > file` | `fs.writeFile()` |
| 目录树 | `find -maxdepth` | `fs.readdir()` 递归 |
| 删除 | `rm -rf` | `fs.rm({ recursive: true })` |
| 重命名 | `mv` | `fs.rename()` |
| 权限 | `chmod` | `fs.chmod()` |
| 打包 | `tar czf` | `child_process.exec('tar')` 或 archiver 库 |

### 5.5 IPC 层设计

本地文件操作需要在 Main 进程执行（`fs` API 不在 Renderer 进程可用）。新增 `local-fs` IPC 通道：

```
Renderer                    Main
  │                          │
  ├─ local-fs-list ─────────►│ fs.readdir + stat
  ├─ local-fs-read ─────────►│ fs.readFile
  ├─ local-fs-write ────────►│ fs.writeFile
  ├─ local-fs-mkdir ────────►│ fs.mkdir
  ├─ local-fs-delete ───────►│ fs.rm
  ├─ local-fs-rename ───────►│ fs.rename
  ├─ local-fs-chmod ────────►│ fs.chmod
  ├─ local-fs-tree ─────────►│ 递归 readdir
  └─ local-fs-stat ─────────►│ fs.stat
```

### 5.6 重点问题

| 问题 | 解决方案 |
|------|---------|
| 权限限制 | 本地 fs API 受当前用户权限限制，无法操作 root 文件。可提示用户但不做 sudo 提权 |
| 大文件读取 | 与 SSH 实现一致，读取时限制大小（如 10MB），超过提示用户 |
| 符号链接 | `fs.stat` 默认跟随链接，`fs.lstat` 不跟随。使用 lstat + 标记 |
| 文件上传/下载 | 本地模式下无意义（源和目标都是本机），隐藏上传/下载按钮 |

## 6. 子系统三：本地系统监控

### 6.1 问题分析

当前监控通过 `sshExecute()` 执行平台特定 Shell 命令采集指标。本地终端可以直接使用 Node.js API，更高效也更可靠。

### 6.2 设计思路

```
           ISystemMonitorService (新增接口)
           ┌───────────────────────────┐
           │ start(intervalMs)         │
           │ stop()                    │
           │ onMetrics(cb)             │
           └─────────────┬─────────────┘
                         │
            ┌────────────┼────────────┐
            ▼                         ▼
  SSHSystemMonitor            LocalSystemMonitor
  Service (现有重命名)         Service (新增)
  ┌────────────────┐       ┌────────────────────┐
  │ sshExecute()   │       │ Main 进程 IPC       │
  │ 解析 Shell 输出 │       │ os.cpus()          │
  │ 计算网速/RTT   │       │ os.freemem()       │
  └────────────────┘       │ os.totalmem()      │
                           │ child_process      │
                           │  (top/ps/df)       │
                           └────────────────────┘
```

### 6.3 本地监控数据源

| 指标 | Node.js API | 回退方案 |
|------|-------------|---------|
| CPU 使用率 | `os.cpus()` 两次采样差值计算 | - |
| CPU 核数 | `os.cpus().length` | - |
| 内存 | `os.freemem()` / `os.totalmem()` | - |
| Swap | 需 Shell 命令：`sysctl vm.swapusage`(mac) / `/proc/meminfo`(linux) | - |
| 磁盘 | `child_process.exec('df -h')` | - |
| 进程列表 | `child_process.exec('ps aux --sort=-%cpu')` | `top -l 1` (mac) |
| 网络流量 | `child_process.exec('netstat -ib')` 两次采样 | `/proc/net/dev` (linux) |
| Uptime | `os.uptime()` | - |
| Ping | 本地无意义，设为 0 或隐藏 | - |

### 6.4 架构选择：Main 进程采集

本地监控在 **Main 进程**执行（`os.*` 和 `child_process` 需要 Node.js 完整能力）：

```
Renderer (插件)              Main 进程
  │                            │
  ├─ local-monitor-start ─────►│ 启动定时器
  │                            │ ┌──────────────────┐
  │                            │ │ 每 3s 采集一次:    │
  │◄─ local-monitor-data ─────│ │ os.cpus()        │
  │                            │ │ os.freemem()     │
  │                            │ │ exec('df -h')    │
  │                            │ │ exec('ps aux')   │
  │                            │ └──────────────────┘
  ├─ local-monitor-stop ──────►│ 停止定时器
```

### 6.5 流程

```
MonitoringSidebar 收到 ConnectionInfo (connectionType='local')
  │
  ▼
创建 LocalSystemMonitorService
  │
  ▼
service.start(3000)
  → IPC: local-monitor-start
  │
  ▼
Main 进程: LocalMonitorCollector
  → 每 3 秒采集 os.cpus / freemem / df / ps
  → 计算 CPU% / 内存% / 网速
  → IPC event: local-monitor-data → Renderer
  │
  ▼
LocalSystemMonitorService.onMetrics(callback)
  → 转换为 SystemMetrics
  → 插件更新 UI
```

### 6.6 重点问题

| 问题 | 解决方案 |
|------|---------|
| Ping 指标无意义 | 本地模式隐藏 Ping，或显示为 "< 1ms" |
| Shell 命令跨平台差异 | 复用现有 `DarwinMonitor` / `LinuxMonitor` / `WindowsMonitor` 的命令构建逻辑 |
| Main 进程性能影响 | `child_process.exec` 是异步的，不阻塞主线程；CPU 采样用 `os.cpus()` 轻量 |
| 多个本地终端 Tab 同时采集？ | 所有本地 Tab 共享同一个 `LocalMonitorCollector` 实例（单例），避免重复采集 |

## 7. 子系统四：IHostConnection + ConnectionInfo

### 7.1 IHostConnection 接口

`IHostConnection` 是能力层的统一入口，组合了终端 I/O、命令执行、文件传输三个基础能力：

| 属性/方法 | 说明 |
|-----------|------|
| `type` | `'ssh' \| 'local'` |
| `connect()` | 建立连接（SSH 连接 / PTY 创建） |
| `disconnect()` | 断开连接 |
| `terminal` | `ITerminalBackend` — 终端数据流 |
| `executor` | `ICommandExecutor` — 命令执行 |
| `transfer?` | `IFileTransfer` — 文件传输（SSH 有，本地为 null） |

### 7.2 两个实现

**SSHHostConnection**：
- `connect()` → 调 `sshService.connect()` 建立 SSH 连接
- `terminal` → `SSHTerminalBackend`
- `executor` → `ElectronShellExecutor`
- `transfer` → 基于 SFTP 的文件传输

**LocalHostConnection**：
- `connect()` → 创建 PTY（可选，也可延迟到 terminal.connect()）
- `terminal` → `LocalTerminalBackend`
- `executor` → `LocalShellExecutor`（独立 PTY）
- `transfer` → `null`（本地无需传输）

### 7.3 HostConnectionFactory

```
HostConnectionFactory.create(host: Host): IHostConnection
  → host.connectionType === 'local' ? new LocalHostConnection()
                                     : new SSHHostConnection(host)
```

### 7.4 TerminalView 使用方式

TerminalView 持有 `IHostConnection`，从中获取各子能力：

```
const connection = HostConnectionFactory.create(host);
await connection.connect();

// 终端 → XTermTerminal
<XTermTerminal backend={connection.terminal} />

// AI Agent → useAIAgent
useAIAgent({ executor: connection.executor })

// 插件 → ConnectionInfo
builtinPluginManager.setConnectionInfo({
  connectionId: connection.terminal.id,
  connectionType: connection.type,
  ...
})
```

### 7.5 插件路由逻辑

插件通过 `ConnectionInfo.connectionType` 路由，结合 `IHostConnection` 暴露的 executor：

```
onConnectionChange(info) {
  if (!info) return;  // 连接断开

  if (info.connectionType === 'local') {
    fileBrowser = new LocalFileBrowserService();
    monitor = new LocalSystemMonitorService(updateCallback);
  } else {
    fileBrowser = new SSHFileBrowserService(info.connectionId);
    monitor = new SSHSystemMonitorService(info.connectionId, updateCallback, ...);
  }
}
```

**原则**：插件代码中不出现 `if (connectionId === 'local')` 字符串判断，统一用 `connectionType` 字段路由。

```
│                     传输层                                │
│                                                         │
│   ssh-service          local-pty-service                 │
│   file-transfer-svc    tunnel-service                   │
│                                                         │
│   （未来迁移至 src/core/）                                │
└─────────────────────────────────────────────────────────┘
```

**依赖方向**：应用层 → 能力层（IHostConnection） → 传输层。
所有箭头向下，无循环依赖。应用层只依赖 `IHostConnection` 暴露的能力接口，不直接访问传输层。

---

## 9. 实现优先级

| 优先级 | 子系统 | 理由 |
|--------|--------|------|
| P0 | ConnectionInfo 扩展 | 所有子系统的路由基础 |
| P1 | AI Agent 本地执行器 | 核心卖点，用户最关心的功能 |
| P2 | 本地文件浏览器 | 使用频率高，但可通过系统文件管理器替代 |
| P3 | 本地系统监控 | 可通过系统自带工具（Activity Monitor）替代 |

---

## 10. 风险与对策

| 风险 | 对策 |
|------|------|
| AI 执行器与用户终端 PTY 冲突 | 独立 PTY 模式，AI 使用单独的 PTY 实例 |
| 本地 fs API 权限不足 | 友好提示，不做 sudo 提权（安全考虑） |
| Main 进程 child_process 阻塞 | 所有命令异步执行，设超时，用 maxBuffer |
| 三个子系统同时开发容易冲突 | 模块独立，接口先行，可并行开发 |
| Windows 路径分隔符差异 | 文件浏览器使用 `path.sep`，统一处理 |
