# 本地终端 SSH 跳转支持方案

> 日期：2026-04-06
> 状态：方案设计

---

## 1. 目标

本地终端用户执行 `ssh user@remote-host` 跳转后，**监控面板、文件浏览器、AI 运维**自动切换到远程主机操作。上层业务代码零改动 — 与远程 SSH 跳转保持完全一致的行为。

## 2. 核心原则

**上层不关心底层是本地还是远程** — `IHostConnection` 接口统一。不管 `LocalHostConnection` 还是 `SSHHostConnection`，上层通过 `fsHandler`、`cmdExecutor`、`monitorCmdExecutor`、`effectiveHostname`、`onHostChanged` 操作，底层自动路由。

## 3. 当前差距

| 能力 | SSHHostConnection (已实现) | LocalHostConnection (待实现) |
|------|--------------------------|----------------------------|
| NestedSSHDetector | ✓ | ✗ |
| ProxyCmdExecutor/ProxyFsHandler | ✓ | ✗ |
| TerminalCmdExecutor (文件浏览) | ✓ | ✗ |
| PrivateShellExecutor (监控) | ✓ 用 SSH extra shell | ✗ |
| effectiveHostname / onHostChanged | ✓ | ✗ |
| onRawData / setMuted | ✓ SSHTerminalBackend | ✗ LocalTerminalBackend |

## 4. 方案设计

### 4.1 核心思路

```
SSH 连接的 PrivateShellExecutor:
  sshCreateShell(connId__monitor)  →  extra shell on jump host
  →  ssh -tt user@target           →  SSH 到目标机
  →  marker 协议执行命令

本地终端的 LocalPrivateShellExecutor:
  localTerminal.create()           →  新建本地 PTY
  →  ssh user@target               →  SSH 到目标机
  →  marker 协议执行命令            ← 完全相同!
```

两者都是：**创建独立的 shell 通道 → SSH 到目标 → marker 协议执行命令**。唯一区别是底层通道创建方式不同（SSH extra shell vs 本地 PTY）。

### 4.2 新增：LocalPrivateShellExecutor

```typescript
// src/core/terminal/LocalPrivateShellExecutor.ts
class LocalPrivateShellExecutor implements ICmdExecutor {
  private _ptyId: string = '';
  private _initialized = false;
  
  constructor(config: {
    sshCommand: string;  // e.g., "ssh dum@access.oa.zego.im"
  }) {}
  
  async execute(command: string): Promise<CmdResult> {
    // Lazy init: 第一次调用时创建 PTY + SSH
    if (!this._initialized) await this._initialize();
    return this._executeInternal(command);
  }
  
  private async _initialize(): Promise<void> {
    // 1. 创建新的本地 PTY
    const { ptyId } = await window.electron.localTerminal.create({ cols: 80, rows: 24 });
    this._ptyId = ptyId;
    
    // 2. 注册数据监听
    this._unsubData = window.electron.localTerminal.onData((id, data) => {
      if (id === this._ptyId) this._dataHandler?.(data);
    });
    
    // 3. 禁用回显
    await window.electron.localTerminal.write(this._ptyId, 'stty -echo\n');
    
    // 4. SSH 到目标机
    await window.electron.localTerminal.write(this._ptyId, this._config.sshCommand + '\n');
    
    // 5. 等待目标 shell ready (检测 Last login / prompt)
    await this._waitForShellReady();
    
    // 6. 在目标机上也禁用回显
    await window.electron.localTerminal.write(this._ptyId, 'stty -echo\n');
  }
  
  // marker 协议执行命令 — 与 PrivateShellExecutor 完全相同
  private _executeInternal(command: string): Promise<CmdResult> { ... }
  
  dispose(): void {
    // 销毁 PTY
    window.electron.localTerminal.destroy(this._ptyId);
  }
}
```

### 4.4 修改：LocalHostConnection

复用 SSHHostConnection 的完整模式：

```typescript
class LocalHostConnection implements IHostConnection {
  // 新增（与 SSHHostConnection 对齐）
  private _proxyFs: ProxyFsHandler;
  private _proxyCmd: ProxyCmdExecutor;
  private _detector: NestedSSHDetector;
  private _monitorExecutor: LocalPrivateShellExecutor | null;
  private _effectiveHostname: string | null;
  
  get fsHandler() { return this._proxyFs; }        // 透明代理
  get cmdExecutor() { return this._proxyCmd; }      // 透明代理
  get monitorCmdExecutor() { return this._monitorExecutor; }
  get effectiveHostname() { return this._effectiveHostname; }
  onHostChanged(cb) { ... }
  
  // SSH 跳转检测后：
  // 1. ProxyCmdExecutor → TerminalCmdExecutor (文件浏览/AI 运维)
  // 2. LocalPrivateShellExecutor 创建新 PTY + SSH (监控)
  // 3. effectiveHostname 更新
}
```

### 4.5 上层代码：零改动

| 模块 | 调用方式 | 行为 |
|------|---------|------|
| 文件浏览器 | `fsHandler.listFiles()` | ProxyFsHandler → TerminalFsHandler → 目标机 |
| 监控面板 | `monitorCmdExecutor.execute()` | LocalPrivateShellExecutor → 独立 PTY → SSH → 目标机 |
| AI 运维 | 写终端 stdin | 命令直接在目标机执行（终端已 SSH 过去） |
| Tab 标题 | `effectiveHostname` | 显示检测到的目标 hostname |

## 5. NestedSSHDetector 改进

本地终端的 oh-my-zsh prompt 重绘导致 ANSI 剥离后文字错乱。需要改进：

### 5.1 问题
```
剥离后: (base) ➜  ~ sssh dum@access.oa.zego.imssh dum@access.oa.zego.im
                       ^^^                       ^^^
                       多了s                     重复拼接
```

### 5.2 改进方向

**方案 A：改进 ANSI 剥离**
- 增强 `stripAnsi` 函数，处理更多终端控制序列（光标移动、行清除等）
- 这些序列导致 oh-my-zsh 的 prompt 重绘内容重复

**方案 B：从用户输入端检测**
- 除了分析终端输出，也监听用户输入（`terminal.write` 调用）
- 用户输入是干净的 `ssh user@host\r`，没有 ANSI 污染
- 配合输出端的 `Last login:` 确认

**推荐方案 B** — 输入端数据干净可靠，不受 prompt 主题影响。
