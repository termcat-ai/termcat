# TermCat 客户端分层重构方案设计

## 一、功能概述

### 目标

将 termcat_client/src 的 Renderer 进程代码按 DDD 领域驱动思想重新分层，建立清晰的依赖方向：

```
plugin → feature → core → base → utils
  上层       ←       调用       →       下层
```

### 核心约束

| # | 约束 | 说明 |
|---|------|------|
| 1 | 严格单向依赖 | 上层可调用下层，**下层绝不能调用上层** |
| 2 | 同层可平行调用 | 同层模块之间允许互相引用 |
| 3 | DDD 领域组织 | 按业务领域（而非技术类型）组织目录 |
| 4 | Main/Preload 不参与 | Electron 主进程 / Preload 保持独立顶层目录 |

### 各层定义

| 层次 | 目录 | 职责 | 关键特征 |
|------|------|------|----------|
| **plugin** | `src/plugins/` | 插件框架 + 内置/第三方插件 | 通过插件 API 访问下层能力，可独立加卸载 |
| **feature** | `src/features/` | 宿主应用的业务功能 | 直接集成在宿主中，组合 core 层能力实现业务 |
| **core** | `src/core/` | 业务基础能力 | 提供领域服务、SDK、业务模型，不含 UI |
| **base** | `src/base/` | 通用基础组件 | 与业务完全无关：网络、日志、配置、i18n |
| **utils** | `src/utils/` | 工具函数 + 基础类型定义 | 纯函数、常量、类型，零依赖 |

## 三、数据模型

### 3.1 各层类型定义归属

类型定义遵循"谁定义、谁使用"原则：

| 类型归属 | 说明 | 示例 |
|----------|------|------|
| **utils/types/** | 全局共享的基础类型，跨层使用 | `Host`, `User`, `Session`, `ViewState`, `ThemeType` |
| **core/领域/types.ts** | 领域内部类型，仅该领域及上层使用 | `AIMessageType`, `ICommandExecutor`, `ITerminalBackend` |
| **feature/领域/types.ts** | 业务功能内部类型 | `AIOpsMessage`, `AIOpsMode`, `AIOpsPanelProps` |
| **plugin/types.ts** | 插件框架类型 | `PluginManifest`, `PluginAPI`, `UIContribution` |

### 3.2 现有类型迁移映射

| 现有文件 | 目标位置 | 说明 |
|----------|----------|------|
| `types/index.ts` | `utils/types/index.ts` | 全局共享基础类型 |
| `types/terminal.ts` | `core/terminal/types.ts` | 终端领域类型 |
| `types/aiOps.ts` | `features/terminal/types.ts` | AI运维业务类型（UI 相关） |
| `types/chatHistory.ts` | `core/chat/types.ts` | 会话记录领域类型 |
| `types/commerce.ts` | `core/commerce/types.ts` | 商业化领域类型 |
| `types/ad.ts` | `core/ad/types.ts` | 广告领域类型 |
| `modules/ai-agent/types.ts` | `core/ai-agent/types.ts` | AI Agent SDK 类型 |

## 四、接口设计

### 4.1 层间通信约定

各层对外暴露能力的方式：

| 层次 | 对外暴露方式 | 消费方式 |
|------|-------------|----------|
| **utils** | 直接 export 函数/类型/常量 | `import { xxx } from '@/utils/...'` |
| **base** | 单例服务 / 工厂函数 / React Context | `import { logger } from '@/base/logger'` |
| **core** | 领域服务类 / 接口 / 事件驱动 | `import { AIAgent } from '@/core/ai-agent'` |
| **feature** | React 组件 / 自定义 Hook | `import { TerminalView } from '@/features/terminal'` |
| **plugin** | 插件 API（PluginContext 注入） | `ctx.terminal.execute(cmd)` |

### 4.2 core 层关键接口

**ai-agent 领域**：保持现有接口不变

| 方法签名 | 用途 |
|----------|------|
| `AIAgent.ask(prompt, mode, config)` | 发起 AI 请求 |
| `AIAgent.confirmExecute(result)` | 确认执行结果 |
| `AIAgent.setExecutor(executor)` | 设置命令执行器 |
| `ICommandExecutor.execute(command, options): Promise<CommandResult>` | 执行命令 |
| `AIAgentConnection.connect(url, token)` | 建立 WebSocket 连接 |

**terminal 领域**：

| 方法签名 | 用途 |
|----------|------|
| `ITerminalBackend.connect(config): Promise<void>` | 连接终端 |
| `ITerminalBackend.write(data): void` | 写入终端 |
| `ITerminalBackend.resize(cols, rows): void` | 调整大小 |
| `IHostConnection.getTerminal(): ITerminalBackend` | 获取终端后端 |
| `IHostConnection.getCmdExecutor(): ICmdExecutor` | 获取命令执行器 |
| `IHostConnection.getFsHandler(): IFsHandler` | 获取文件系统处理器 |
| `TerminalBackendFactory.create(host): ITerminalBackend` | 工厂创建 |
| `HostConnectionFactory.create(host): IHostConnection` | 工厂创建 |

**ssh 领域**：

| 方法签名 | 用途 |
|----------|------|
| `SSHManager.connect(config): Promise<string>` | 创建 SSH 连接 |
| `SSHManager.disconnect(sessionId): void` | 断开连接 |
| `SSHConfigParser.parse(configPath): HostConfig[]` | 解析 SSH 配置 |

**commerce 领域**：

| 方法签名 | 用途 |
|----------|------|
| `commerceService.fetchConfig(): Promise<CommerceConfig>` | 获取商业化配置 |
| `paymentService.createOrder(plan): Promise<Order>` | 创建订单 |

### 4.3 base 层关键接口

| 模块 | 方法签名 | 用途 |
|------|----------|------|
| http | `api(path, params): Promise<Response>` | 统一 HTTP 请求 |
| http | `apiService.get/post(url, data)` | HTTP 方法 |
| logger | `logger.info(module, event, msg, data)` | 日志记录 |
| logger | `logger.withFields(fields): ScopedLogger` | 创建模块日志 |
| i18n | `useI18n(): { t, language, setLanguage }` | 国际化 Hook |
| storage | `localStorageService.get/set(key, value)` | 本地存储封装 |

### 4.4 plugin 层接口

插件通过 `PluginContext` 访问宿主能力，**不直接 import core/feature 模块**：

| 接口 | 用途 |
|------|------|
| `ctx.ui.registerPanel(location, component)` | 注册 UI 面板 |
| `ctx.ui.registerToolbarAction(action)` | 注册工具栏按钮 |
| `ctx.terminal.onData(sessionId, callback)` | 监听终端数据 |
| `ctx.terminal.execute(sessionId, command)` | 执行命令 |
| `ctx.fs.list(sessionId, path)` | 列出文件 |
| `ctx.fs.read/write(sessionId, path, data)` | 读写文件 |
| `ctx.i18n.t(key)` | 国际化 |
| `ctx.events.emit/on(event, data)` | 事件通信 |

### 5.3 msg-viewer 的特殊定位

msg-viewer 是一个**与业务逻辑完全解耦的通用 UI 控件**，被 plugin 层（ai-ops）和潜在的 feature 层使用。它不属于任何一层，而是作为**跨层共享组件**独立存在。

放在 `src/shared-components/` 下，与分层体系平行，各层均可引用：

```
shared-components/msg-viewer  ←── plugin/builtin/ai-ops 引用
                               ←── feature/terminal 可引用
                               ──→ 自身只依赖 utils + base（i18n）
```

## 七、关键难点及解决方案

### 难点 1：循环依赖和逆向依赖检测

**问题**：重构后如何确保没有下层调用上层的情况？纯靠人工 code review 不可靠。

**解决方案**：引入 ESLint 规则 `eslint-plugin-import` + `eslint-plugin-boundaries`，配置分层依赖规则：

- 定义层级 zones：utils, base, core, features, plugins
- 配置每个 zone 允许 import 的目标 zone
- CI 中运行 lint 检查，违反即报错

规则示例：
- `plugins` 可引用 `features, core, base, utils, shared-components`
- `features` 可引用 `core, base, utils, shared-components`
- `core` 可引用 `base, utils`
- `base` 可引用 `utils`
- `utils` 不可引用任何其他层

### 难点 2：authService 的层级归属

**问题**：`authService` 包含 token 管理（base 级别）和用户信息管理（core 级别），职责跨层。

**解决方案**：拆分为两部分：
- `base/http/api.ts` 中的拦截器负责 token 注入 / 401 处理（纯通信层面）
- `core/auth/authService.ts` 负责登录/登出、用户信息、token 存取（业务语义）

api.ts 通过**依赖注入/回调**方式获取 token，而非直接 import authService：
- api.ts 暴露 `setTokenProvider(fn)` 方法
- authService 初始化时调用 `api.setTokenProvider(() => this.getToken())`

这样 base 层不依赖 core 层。

### 难点 3：aiWebSocketService 的定位

**问题**：`aiWebSocketService.ts` 当前兼具 WebSocket 通信基础能力和 AI 业务语义。

**解决方案**：
- 如果它仅做 WebSocket 连接池管理（建连/断连/心跳），放 `base/websocket/`
- AI 协议解析和业务消息处理由 `core/ai-agent/AIAgentConnection.ts` 负责
- 当前代码中 aiWebSocketService 已被标注为"兼容层"且委托给 AIAgentConnection，可在重构中逐步废弃，直接使用 core 层 AIAgentConnection

### 难点 4：feature 层 hooks 对 core 层的依赖注入

**问题**：`useAIAgent` hook 需要创建 `AIAgent` 实例并绑定 `ElectronShellExecutor`，这涉及 core 层多个模块的组装。

**解决方案**：在 feature 层 hook 中组装 core 层能力是合理的，这正是 feature 层的职责——编排核心能力形成业务功能。hook 可以 import core 层的类和工厂方法来完成组装。关键是 core 层的 AIAgent、Executor 等不反向依赖 feature 层。

### 难点 5：渐进式迁移保持系统可用

**问题**：大规模文件移动会导致大量 import 路径失效，一次性迁移风险大。

**解决方案**：
- 采用"创建新 → re-export 旧 → 逐步切换 → 删除旧"策略
- 每个阶段迁移后立即运行 `npm run build` 验证
- 利用 tsconfig paths 别名，旧路径可临时 re-export 到新路径
- 每个阶段独立一个 git commit，便于 bisect 和回滚

### 难点 6：msg-viewer 的层级归属

**问题**：msg-viewer 是通用 UI 控件，被 plugin 层使用，但它自身有 UI 组件（属于展示层）又有类型定义。不适合放在任何一个业务层中。

**解决方案**：作为 `shared-components/` 独立存在，和分层体系平行。它只允许依赖 `utils` 和 `base`（i18n），不依赖任何业务层。任何层都可以引用它。这类似于一个内部"UI 库"。

### 难点 7：插件如何安全访问宿主能力

**问题**：内置插件当前直接 import features/core 层代码，破坏了插件的独立性。

**解决方案**：
- 内置插件作为"特权插件"，暂时允许直接 import core 层（和 feature 层中通过 context 暴露的能力）
- 长期目标是通过 `PluginContext` 统一注入能力，实现完全解耦
- 重构第一阶段不改变内置插件的引用方式，仅更新 import 路径
