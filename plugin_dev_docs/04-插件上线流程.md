# TermCat 插件上线流程

> 本文档描述 TermCat 内置插件和外部插件的打包、发布和上线流程。

---

## 一、内置插件上线

内置插件作为应用的一部分，随应用版本一起发布，无需独立上线流程。

### 上线步骤

1. **开发完成** — 代码合并到主分支
2. **懒加载注册** — 确认已在 `plugins/builtin/index.ts` 中使用动态 `import()` 注册
3. **事件常量抽取** — 确认跨插件事件已定义在 `plugins/builtin/events.ts`
4. **国际化完成** — 确认 `locales/` 下 zh/en/es 三种语言均已翻译
5. **构建验证** — 运行 `npm run build`，检查 `dist/bundle-report.html` 确认：
   - 插件代码在独立 chunk 中，未污染主包
   - 主包大小未显著增长
6. **功能测试** — 开发环境验证插件功能
7. **随版本发布** — 跟随应用版本号发布

### 上线检查清单

- [ ] 插件 ID 以 `builtin-` 开头
- [ ] `index.ts` 导出符合 `BuiltinPlugin` 接口
- [ ] 实现了 `getLocalizedName` 和 `getLocalizedDescription`
- [ ] 底部面板实现了 `getLocalizedTitle`
- [ ] `locales/` 下有 zh.ts、en.ts、es.ts
- [ ] 没有硬编码的中文/英文文本
- [ ] 组件中使用 `useT()` 获取翻译
- [ ] 在 `plugins/builtin/index.ts` 中使用 `import()` 动态加载
- [ ] 宿主代码没有直接 import 插件模块（使用 events.ts 中的事件常量）
- [ ] 使用 CSS 变量（`var(--text-primary)` 等）而非硬编码颜色
- [ ] `deactivate()` 或 `subscriptions` 正确清理定时器、监听器等资源
- [ ] Bundle report 确认插件在独立 chunk

## 二、外部插件上线

### 2.1 开发与打包

#### 1. 编译

```bash
# 安装依赖
npm install

# 编译 TypeScript
npx tsc

# 验证产物
ls dist/extension.js
```

#### 2. 验证 package.json

确认 `termcat` 配置段完整：

```json
{
  "name": "termcat-plugin-xxx",
  "version": "1.0.0",
  "termcat": {
    "displayName": "插件显示名称",
    "description": "插件功能描述",
    "entry": "dist/extension.js",
    "activationEvents": ["onStartup"],
    "permissions": ["terminal.read"]
  }
}
```

**必要字段检查：**

| 字段 | 必须 | 说明 |
|------|------|------|
| `name` | 是 | 作为插件 ID（npm 包名格式） |
| `version` | 是 | 语义化版本号 |
| `termcat.displayName` | 是 | 用户可见名称 |
| `termcat.description` | 是 | 功能描述 |
| `termcat.entry` | 是 | 入口文件路径（相对于插件根目录） |
| `termcat.activationEvents` | 是 | 至少一个激活事件 |
| `termcat.permissions` | 建议 | 声明所需权限 |

#### 3. 打包为 .tgz

```bash
# 方式一：npm pack（推荐）
npm pack
# 产生: termcat-plugin-xxx-1.0.0.tgz

# 方式二：手动打包
tar -czf termcat-plugin-xxx-1.0.0.tgz \
  package.json \
  dist/
```

**打包文件结构要求：**

```
termcat-plugin-xxx-1.0.0.tgz
└── (解压后)
    ├── package.json          # 必须在根目录
    └── dist/
        └── extension.js      # 入口文件
```

> **注意：** `npm pack` 会自动将文件放在 `package/` 子目录下。TermCat 安装器会自动处理这种结构，使用 `tar -xzf` 解压到目标目录。

### 2.2 本地测试

#### 方式一：手动安装

将 `.tgz` 解压到插件目录进行测试：

```bash
# macOS
PLUGIN_DIR=~/Library/Application\ Support/TermCat/plugins/termcat-plugin-xxx
mkdir -p "$PLUGIN_DIR"
tar -xzf termcat-plugin-xxx-1.0.0.tgz -C "$PLUGIN_DIR"

# 重启 TermCat 验证
```

#### 方式二：通过设置页安装

1. 将 `.tgz` 文件放到本地 HTTP 服务器或文件分享服务
2. 在 TermCat「设置 → 插件」页面输入 URL 安装
3. 安装后自动激活（如果 `activationEvents` 匹配）

#### 验证要点

- [ ] 插件出现在「设置 → 插件」列表中
- [ ] 显示名称和描述正确
- [ ] 插件状态为 `activated`
- [ ] 注册的 UI 元素（状态栏、工具栏、面板等）正常显示
- [ ] 功能正常工作
- [ ] 禁用/启用切换正常
- [ ] 卸载后完全清理

### 2.3 发布

#### 托管方式

TermCat 外部插件通过 URL 分发，支持以下托管方式：

| 方式 | 适用场景 | 示例 URL |
|------|---------|---------|
| 自建服务器 | 企业内部 | `https://plugins.company.com/termcat-plugin-xxx-1.0.0.tgz` |
| GitHub Release | 开源插件 | `https://github.com/user/repo/releases/download/v1.0.0/termcat-plugin-xxx-1.0.0.tgz` |
| npm Registry | npm 生态 | 需要额外适配 |
| OSS/CDN | 大规模分发 | `https://cdn.example.com/plugins/termcat-plugin-xxx-1.0.0.tgz` |

#### 发布到 GitHub Release

```bash
# 1. 创建 tag
git tag v1.0.0
git push origin v1.0.0

# 2. 创建 Release 并上传 .tgz
gh release create v1.0.0 termcat-plugin-xxx-1.0.0.tgz \
  --title "v1.0.0" \
  --notes "Initial release"
```

### 2.4 版本更新

更新插件时：

1. 修改 `package.json` 中的 `version` 字段
2. 重新编译和打包
3. 上传新版本的 `.tgz`
4. 用户在设置页重新安装即可（会自动清理旧版本目录）

## 三、插件审核标准（建议）

### 安全性

- [ ] 不包含恶意代码
- [ ] 权限声明合理（最小权限原则）
- [ ] 不在 `activate()` 中执行破坏性操作
- [ ] 不读写非授权路径的文件
- [ ] 不在未声明 `network` 权限时发起网络请求

### 稳定性

- [ ] `activate()` 和 `deactivate()` 不抛出未捕获异常
- [ ] 定时器、监听器等资源通过 `subscriptions` 正确清理
- [ ] 不阻塞 Main 进程事件循环
- [ ] 错误场景有容错处理

### 兼容性

- [ ] 入口文件为 CommonJS 格式（`module.exports` / `exports.activate`）
- [ ] 不依赖 Electron 内部 API
- [ ] 不依赖特定操作系统的二进制文件（或声明平台兼容性）
- [ ] Node.js API 使用版本与 Electron 内置版本兼容

### 用户体验

- [ ] `displayName` 和 `description` 清晰易懂
- [ ] 状态栏/工具栏等 UI 不过度占用空间
- [ ] 通知使用恰当的级别（不滥用 `error` 级别）
- [ ] 模板面板数据格式正确，不导致渲染异常

## 四、故障排查

### 插件安装失败

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| 下载失败 | URL 不可访问或重定向次数超过 5 次 | 检查 URL 可访问性 |
| 解压失败 | .tgz 格式不正确 | 使用 `npm pack` 重新打包 |
| 未发现插件 | package.json 缺少 `termcat` 字段 | 添加 `termcat` 配置段 |
| 激活失败 | 入口文件不存在 | 确认 `termcat.entry` 路径正确 |
| 激活失败 | 未导出 `activate` 函数 | 确认入口文件 `exports.activate = function...` |

### 插件运行异常

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| API 调用报权限错误 | 未声明所需权限 | 在 `permissions` 中添加 |
| UI 不更新 | 面板数据格式不匹配模板类型 | 检查 `SectionDescriptor` 格式 |
| 面板 Renderer reload 后消失 | 外部插件面板数据未缓存 | 框架自动处理，检查 `panelRegistrations` 缓存 |
| 停用后资源泄漏 | 未使用 `subscriptions` 管理资源 | 所有 `Disposable` 推入 `context.subscriptions` |

### 日志查看

外部插件日志前缀为 `[Plugin:<pluginId>]`，在 Electron 的 Main 进程控制台查看：

```bash
# macOS 开发模式
# 通过 Electron DevTools 的 Main 进程 Console 查看

# 日志文件位置
# macOS: ~/Library/Logs/TermCat/termcat.log
# Windows: %APPDATA%/TermCat/logs/termcat.log
# Linux: ~/.config/TermCat/logs/termcat.log
```

## 五、完整上线流程总结

```
开发 → 编译 → 打包(.tgz) → 本地测试 → 审核 → 上传托管 → 用户安装
                                                    ↓
                                              URL 分发给用户
                                                    ↓
                                      设置 → 插件 → 输入 URL 安装
```

### 外部插件快速上线命令

```bash
# 一键打包 + 创建 GitHub Release
npm run build && npm pack && \
gh release create v$(node -p "require('./package.json').version") \
  *.tgz --generate-notes
```
